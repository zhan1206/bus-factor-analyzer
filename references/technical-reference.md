# Bus Factor Analyzer - Technical Reference

## 核心算法

### Bus Factor 计算

Bus Factor 表示"如果这些关键人物离开，项目会陷入困境"的人数。

**计算方法**：
1. 从 git log 获取所有贡献者的提交记录
2. 按代码行数（added + deleted）统计每个贡献者的贡献量
3. 按贡献量降序排列
4. 累加直到达到 50% 和 80% 阈值，记录所需人数

### Gini 系数

衡量贡献分布的不均匀程度（0 = 完全均匀，1 = 一人独占）：

```
Gini = (2 * sum(i * x_i) - (n+1) * sum(x_i)) / (n * sum(x_i))
```

其中 x_i 是排序后的贡献量。

### 风险等级

| Bus Factor | 风险等级 | 说明 |
|------------|----------|------|
| 1 | CRITICAL | 极度危险，项目完全依赖单一贡献者 |
| 2 | HIGH | 高风险，缺乏核心团队深度 |
| 3-4 | MEDIUM | 中等风险，有一定缓冲 |
| 5+ | LOW | 低风险，团队健康 |

## 局限性

- 无法检测提交 authorship 问题（rebase/squash 后历史可能不准确）
- 代码行数不等于代码价值（注释行也计入）
- 不考虑贡献的时间分布（早期贡献者可能更重要）
- 需要仓库有实际的 git 历史

## 扩展方向

- 结合 CODEOWNERS 文件分析关键文件所有权
- 考虑 issue/PR 评论者的隐性贡献
- 跟踪随时间变化的 Bus Factor 趋势
- 与 CI/CD 配置关联分析
